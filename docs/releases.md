# Releases

## Versioning

Releases follow [Semantic Versioning](https://semver.org/) and are tagged
**without** a `v` prefix. For example, version `0.0.18` corresponds to
git tag `0.0.18` and container image tag
`ghcr.io/observiq/bindplane-operator:0.0.18`.

## Container Image Signing

Every release container image is signed using
[cosign](https://docs.sigstore.dev/cosign/overview/) keyless signing. The
signature is produced in GitHub Actions using an OIDC identity token issued
by GitHub, so there are no long-lived signing keys to manage or rotate.

A valid signature proves that the image was built by the
`observiq/bindplane-operator` GitHub Actions release workflow and has not
been tampered with.

### Verifying the signature

```bash
cosign verify \
  --certificate-oidc-issuer https://token.actions.githubusercontent.com \
  --certificate-identity-regexp '^https://github\.com/observiq/bindplane-operator/' \
  ghcr.io/observiq/bindplane-operator:0.0.18
```

A successful verification prints the signing certificate details and
Rekor transparency log entry. A failed verification exits non-zero with
an error message.

## Software Bill of Materials (SBOM)

An SBOM is a machine-readable inventory of every dependency (libraries,
modules, packages) included in a software artifact. SBOMs enable
consumers to audit the supply chain, track vulnerabilities, and satisfy
compliance requirements.

### Binary SBOMs

Each GitHub release includes SBOM files for the compiled binaries in
[SPDX](https://spdx.dev/) JSON format:

- `manager_linux_amd64.sbom.json`
- `manager_linux_arm64.sbom.json`

These are generated by [syft](https://github.com/anchore/syft) during the
release build. You can download them from the
[GitHub releases page](https://github.com/observiq/bindplane-operator/releases)
and inspect them with any SPDX-compatible tool:

```bash
# Download the SBOM for a specific release
gh release download 0.0.18 \
  --repo observiq/bindplane-operator \
  --pattern 'manager_linux_amd64.sbom.json'

# Inspect with syft (or any SPDX JSON reader)
cat manager_linux_amd64.sbom.json | jq '.packages[] | .name' | head
```

### Container Image SBOM Attestations

In addition to the binary SBOMs, each release container image has an SBOM
attached as an [in-toto](https://in-toto.io/) attestation. This attestation
is signed with cosign and stored alongside the image in the container
registry, so consumers can verify both the image signature and its SBOM in
a single workflow.

#### Verifying the SBOM attestation

```bash
cosign verify-attestation \
  --certificate-oidc-issuer https://token.actions.githubusercontent.com \
  --certificate-identity-regexp '^https://github\.com/observiq/bindplane-operator/' \
  --type spdxjson \
  ghcr.io/observiq/bindplane-operator:0.0.18
```

#### Extracting the SBOM from the attestation

```bash
cosign verify-attestation \
  --certificate-oidc-issuer https://token.actions.githubusercontent.com \
  --certificate-identity-regexp '^https://github\.com/observiq/bindplane-operator/' \
  --type spdxjson \
  ghcr.io/observiq/bindplane-operator:0.0.18 \
  | jq -r '.payload' | base64 -d | jq '.predicate'
```

## Checking Supply Chain Artifacts

To see all supply chain artifacts (signatures and attestations) attached to
an image:

```bash
cosign tree ghcr.io/observiq/bindplane-operator:0.0.18
```
